## 一句话定位

Aftelle 是 `.digital_resident` 数字居民文件的桌面运行容器。

平台方向:**先做 macOS,Apple 生态优先**(后续 iOS / iPad / Vision Pro 顺延)。Windows / Android 为远期分支(大脑可搬、身体重做)。AR 身体放到 Stage 8。

---

## 第七阶段核心目标

1. 启动 Aftelle Desktop
2. 显示默认粒子 Shell
3. 导入 `.digital_resident` 文件
4. 加载第一个数字居民
5. 形成粒子生命体主 UI
6. 支持文字 / 语音对话
7. 支持基础设置
8. 支持屏幕指导模式雏形
9. 开始开发编排系统
10. 为后续双居民交互预留接口

**重点:** 第七阶段主线仍是单居民运行体验,双居民交互设计保留,但放在后半段(见03_dev_plan.md 7.7/7.8)。

## 1. 启动设计

用户双击 Aftelle 图标后:
```
图标粒子化 → 灰白粒子飘入屏幕中央 → 形成不规则圆形粒子点阵
→ 中间短暂浮现 Aftelle → App 名称消散 → 粒子点阵进入呼吸状态
```

默认形态叫 **Aftelle Shell**。它不是正式数字居民,而是运行容器的默认壳。

---

## 2. 默认 Shell 状态

启动后,中央是灰白色圆形不规则粒子点阵。它可以:简单问候用户、提示导入 `.digital_resident`、响应鼠标靠近、通过字幕基础交流、进入设置界面。

示例:"我还没有载入数字居民文件。你可以导入一个 .digital_resident,让我开始运行它。"

---

## 3. 导入第一个数字居民

用户导入 `.digital_resident` 后:
```
Aftelle Shell 暂停呼吸 → 粒子向中心坍缩 → 校验 DR 文件
→ 读取身份/颜色/外观/运行策略 → 圆形点阵重组成该居民的圆形/抽象粒子生命体
→ 形成第一个正式数字居民
```

默认规则:
- 第一个导入的居民 = 当前主居民
- 主居民占据中央主视觉位置,拥有主要对话权
- 主居民颜色优先用文件内置颜色;文件无颜色则默认金色/暖白

这里的"主"不是人格等级,而是**当前交互焦点**。

---

## 3.1 抽象半身粒子 Avatar

**抽象半身粒子 Avatar(Abstract Bust Avatar)** 是在粒子生命体基础上增加抽象头部、发型、五官、肩颈和上胸轮廓,用于增强数字居民活人感与人格识别,但不是写实数字人系统。

Stage 7 节奏:
- 7.3 只做 `particle_core` 默认形态、`avatar_mode: particle_core / abstract_bust` 本地 UI / 渲染层预留、渲染切换接口和字幕基础。
- 7.4 承接抽象半身人格轮廓设计:抽象头部、发型、五官、肩颈、上胸,以及 masculine / feminine / neutral 视觉气质 preset。
- 7.5 承接 speaking 时的嘴部粒子脉冲 / 轻量口部同步。

允许范围:
- 圆形粒子形态与抽象半身形态自由切换。
- idle / thinking / speaking / sleeping / error 五种状态表现。
- 情绪驱动眉眼嘴轻微变化,但情绪来源必须是 RuntimeCore 返回的 `visual_state` / `resident_state`。
- speaking 时嘴部粒子脉冲。
- Debug Panel 可显示 `avatar_mode` / `visual_state`。

禁止范围:
- 写实人脸、真实皮肤材质、完整身体。
- 骨骼系统、Blendshape、精准 lip sync、viseme。
- 牙齿、舌头、真实口腔、服装系统。
- Avatar 编辑器、ARKit / RealityKit 正式身体、3D 数字人系统。
- Aftelle 推理人格或情绪、修改 DR schema、修改 Runtime API、直连 TTS Provider。

## 4. 主次关系规则

第七阶段采用:主次按导入顺序决定,第一个导入=主居民,第二个=次居民,后续可手动调换。

产品规则:
- 主居民:默认回应用户、位于主视觉中心、字幕优先显示
- 次居民:辅助补充、位于侧方或后方、低亮度待机
- 用户可在设置中切换主次,也可直接指定某位居民回答

注意:主次只是运行会话中的交互优先级,不代表身份高低。

双居民设计保留,但属于 Stage 7 Extended Demo,不作为 7.1–7.5 单居民 MVP 的第一优先级。落地节奏**以03_dev_plan.md v5 为准**:

- 次居民:侧方/后景/粒子密度低
- 谁说话谁发光
- 两者互动时出现粒子桥
- 可手动调换左右位置和主次关系

## 7. 对话 UI

不要做普通聊天软件。建议:
- 底部:电影式字幕
- 底部隐藏:文字输入框
- 文字输入为主;语音输入 Stage 7 只预留(长按或快捷键),不作为必做
- 右上角:运行状态
- 右侧:设置抽屉
- 隐藏入口:Debug / Trace

字幕规则:用户输入→底部小字幕;居民回复→底部中央电影字幕;系统状态→右上角极简提示(如 Resident loaded / Thinking / Memory updated / Screen guide enabled)。

### 7.1 表现映射边界

产品上可以定义每个居民不同的情绪表现,但映射必须来自 `.digital_resident` 的 per-resident 数据。比如同样是 concerned,某个居民可以粒子收束、声音放轻、回复前停顿;另一个居民可以亮度降低、语速变慢。Aftelle kernel 只执行通用机制,不得把具体居民的口吻、节奏、颜色、停顿写死进 kernel。

这一条的权威边界见 `aftelle_runtime_boundary.md §3`。

### 7.2 记忆边界为什么要分开

多居民体验里必须区分三种记忆边界:
- Private Memory:居民自己的长期记忆,不被其他居民偷看。
- Shared Session Context:当前会话共享上下文,用于当场协作。
- Public Transcript:用户可见的对话记录,用于回看和解释。

这样做是为了避免串人格、串关系、串隐私。两个居民可以在同一场对话里协作,但不能因为协作就把彼此的私有记忆合并成一团。

### 7.3 诚实性边界

居民要有生命感,但"是否、何时、如何说明自己不是人"必须是显式设计参数。去 AI 味是去掉套话和模板感,不是隐藏身份或模糊披露。自我披露策略需要随居民设计一起被审视,不能默认消失。

## 8. 设置设计

保留"神经元 / 星图设置"概念,但 **Stage 7 先做传统设置抽屉(Settings UI MVP),星图设置降为 Stage 8 或 7.11 录屏彩蛋**。

设置入口:
- 鼠标滚轮放大粒子核心 → 进入神经元星图设置
- 点击粒子核心 → 打开设置星环
- Cmd / Ctrl + , → 打开传统设置抽屉

设置分类:模型、记忆、声音、外观、权限、屏幕指导、文件管理、Debug / Trace。

高级感来自神经元星图;可用性靠传统设置抽屉兜底。

---

## 9. 屏幕指导模式

第七阶段只做雏形,不做完全控制电脑。

流程:
```
用户:帮我找到导出按钮 → 居民缩小到左下角圆形粒子核心
→ 粒子光标飞向目标区域 → 目标位置出现粒子环 → 字幕提示用户操作
```

原则:默认只指导;不自动点击;自动操作必须二次确认;必须走权限系统。

---

## 10. 退出动画

用户语音或文字要求退出:
```
用户:退出 Aftelle → 居民确认 → 保存状态/记忆
→ 粒子逐渐发散 → 粒子回流 Dock/任务栏图标 → App 关闭
```

如果未保存:"还有未保存的运行状态,需要保存后再离开吗?"

Aftelle Desktop macOS 壳、启动粒子动画、默认 Aftelle Shell、导入 `.digital_resident`、DR 校验与加载、第一个数字居民圆形/抽象主视觉形态、文字对话、TTS 输出(语音输入仅预留)、传统设置抽屉、Runtime Trace 面板、编排系统薄壳。

**注:屏幕指导属于 7.9 后段原型,不属于 7.1–7.5 单居民闭环;星图设置非必做。**

**后半段做(对应03_dev_plan.md 7.6–7.8):**
行业居民基础版、导入第二个居民、主次按导入顺序生成、主次手动调换、双居民同屏、双居民轮流对话、双居民粒子互动、编排系统调度双居民。

**第七阶段不做:**
AR 相机、移动端 App、写实真人形象、完整身体、骨骼 / Blendshape / 精准 lip sync、Avatar 编辑器、3D 数字人、无限多居民、社交系统、市场系统、云同步、完全自动控制电脑。

## 最终版本总结

第七阶段的 Aftelle Desktop 先完成一个可信的数字居民运行容器:启动后以灰白粒子 Shell 出现,导入第一个 `.digital_resident` 后转化为主数字居民,支持对话、基础设置、粒子情绪表达与最小 TTS/字幕。双居民交互、屏幕指导雏形属于 Extended Demo 后半段能力(03_dev_plan.md 7.7/7.8/7.9),不作为 MVP 基线;编排系统从 7.1 开始以薄壳形式开发,先服务单居民闭环。
